Based on the codebase, here is a detailed breakdown of the main flow, how an EU document is processed, where the edge case engine fits in, and how deduplication is handled.

1. Document Entry & Initial Routing
When an EU document comes in, the process starts at index.py which triggers EUService (service/base/EUService.py).

Routing: The service checks if it's running in RECON_MODE (Human-in-the-loop) or normal real-time mode. For standard ingestion, it initializes EURtProcess (service/processors/eu_processors.py).
Document Type: Depending on Config.EXTRACT_HTML, it delegates the raw content to either html_extractor or pdf_extractor.
2. Main Extraction Flow (eu_data_point_extractor_utils.py)
Assuming HTML extraction, the document is broken down into rows/tables. The content is processed asynchronously using parallelized_processing.

LLM Extraction: process_content makes the first LLM call (get_first_call_output) to extract entities. It uses context limit checks and pagination (curate_intermediate_output_list) to ensure all data is captured.
Input: Raw HTML strings/table chunks.
Output: A raw JSON list of extracted entities (dictionaries).
3. Name Adjustment Pipeline (The Hook)
Once the LLM returns a batch of entities, they are immediately passed to get_name_adjusted_records(temp_output_list) before being appended to the final results list. This function acts as the central hub for name processing.

It sequentially runs:

adjust_for_names(temp_output_list)
permute_names(temp_output_list)
run_name_engine_v2_pipeline(raw_names)
4. Edge Case Use: When, Where, and How
Immediately after the standard permutations are complete, the Edge Case Gate (apply_edge_case_gate in ec_engine.py) is triggered.

When & Where: It runs as the final step inside get_name_adjusted_records for every batch of extracted entities.
The Gate: For each entity, it performs a similarity search against a reference file (edge_case_reference.json). If the primary_name has a similarity score >= 0.90 against a known complex reference, the edge case flow is triggered.
Input to Edge Case:
complex_name: The raw, complex sanctioned entity name.
primary_name: The cleaned primary name.
context: The full entity dictionary (including completeInformation and other metadata).
Execution: It calls run_edge_case_flow_parallel(), which fires the edge case prompt in a background thread so it doesn't block the extraction of other entities. The edge case LLM call uses the top similar reference examples to generate permutations.
Output from Edge Case: The LLM returns a dictionary containing new aliases (Latin script), aliases_og (non-Latin script), and reasoning.
5. What Happens After the Edge Case? (Merging & Deduplication)
After submitting all eligible entities to the background edge case threads, apply_edge_case_gate enters "Phase 2".

It waits to collect the results from all futures (with a 120-second timeout).
Deduplication happens right here using the _merge_and_dedup helper function.
How Deduplication is done:
It takes the existing aliases (generated by the main flow) and the new_items (generated by the edge case LLM).
It converts everything to lowercase and strips whitespace to do a case-insensitive comparison.
It checks if the edge-case alias already exists in the existing list. If not, it appends it.
It explicitly filters out the primary_name itself so it doesn't accidentally end up in the aliases list.
6. Final Steps
After the edge case gate returns the deduplicated and enriched temp_output_list, the entities are aggregated. Finally, the list of all entities is sent to Translator (eu_translator.py), which converts them to the standard CDF schema. The final JSON payload is uploaded to S3 and a success notification is published via NATS for downstream ingestion.

📝 The Scenario
Let’s say a new EU Sanctions Press Release (HTML document) arrives. It contains the following text in a table row:

"1. Abu Al-Hussain (alias Abu Hussain, Abu Al-Hasan; DOB: 1980-01-01). Designated for activities in Syria."

Step 1: Document Entry (EUService & EURtProcess)
The system receives the document, checks the configuration, and routes it to the HTML extractor.

Action: The raw HTML table is parsed using BeautifulSoup and split into chunks.
Example Input: <tr><td>1. Abu Al-Hussain (alias Abu Hussain, Abu Al-Hasan... </td></tr>
Step 2: Main LLM Data Extraction
The text chunks are sent to the primary LLM (e.g., GPT-4) in parallel to extract structured data points.

Action: The LLM reads the text and maps it to a JSON dictionary based on prompt instructions.
Example Output: The LLM outputs a temporary list of entity dictionaries:
json
[
  {
    "primary_name": "Abu Al-Hussain",
    "name_info": "Abu Al-Hussain (alias Abu Hussain, Abu Al-Hasan)",
    "dob": "1980-01-01",
    "completeInformation": "1. Abu Al-Hussain (alias Abu Hussain..."
  }
]
Step 3: Name Adjustment & Permutation (The Hook)
Before the entity is saved, it is passed through get_name_adjusted_records. This function triggers a sequence of standard scripts to clean up the names.

Action: adjust_for_names() and permute_names() parse the name_info string to extract explicit aliases.
Example Output: The dictionary is updated with basic aliases extracted from the text:
json
{
  "primary_name": "Abu Al-Hussain",
  "aliases": ["Abu Hussain", "Abu Al-Hasan"],
  "aliases_og": []
}
Step 4: The Edge Case Gate (apply_edge_case_gate)
Immediately after standard permutations, the system checks if this name is complex enough to require the Edge Case LLM.

Similarity Search: The system compares "Abu Al-Hussain" against a pre-loaded reference file (edge_case_reference.json) using SequenceMatcher.
The Gate Condition: It finds a match (e.g., a reference rule for "Abu/Al" Arabic naming conventions) with a similarity score of 0.95 (which is >= the 0.90 threshold).
Parallel Trigger: Because it passed the gate, the system fires run_edge_case_flow_parallel() in a background thread so the main extraction loop isn't paused.
What happens inside the Edge Case Engine?

Prompt Building: It takes the top 5 similar reference examples and builds a specialized prompt.
LLM Call: It asks the LLM: "Given this complex name 'Abu Al-Hussain (alias Abu Hussain, Abu Al-Hasan)', generate all possible structural, phonetic, and non-Latin aliases."
Edge Case Output: The background thread returns:
json
{
  "aliases": ["Abou Al-Hussain", "Abu Al-Hasan", "Abu al Husain"],
  "aliases_og": ["أبو الحسين"],
  "reasoning": "Generated standard Arabic transliterations and phonetic variants."
}
Step 5: Merging and Deduplication
The main thread waits (up to 120 seconds) for the background edge case thread to finish. Once it gets the result, it merges it with the data from Step 3 using the _merge_and_dedup function.

Action: It combines the original aliases (["Abu Hussain", "Abu Al-Hasan"]) with the edge-case aliases (["Abou Al-Hussain", "Abu Al-Hasan", "Abu al Husain"]).
How Deduplication Works:
All strings are .strip().lower() for case-insensitive comparison.
"Abu Al-Hasan" was generated by both the standard script and the Edge Case LLM. The system sees it already exists and drops the duplicate.
It ensures the primary_name ("Abu Al-Hussain") isn't accidentally added as an alias.
Example Output (Final combined list):
json
{
  "primary_name": "Abu Al-Hussain",
  "aliases": ["Abu Hussain", "Abu Al-Hasan", "Abou Al-Hussain", "Abu al Husain"],
  "aliases_og": ["أبو الحسين"]
}
Step 6: Translation to CDF & Publishing
With the entity completely enriched and deduplicated, the process finishes.

Action: The entity is passed to Translator (eu_translator.py), which maps the keys to the final Common Data Format (CDF) schema required by the downstream database.
Final Destination: The CDF JSON is saved to AWS S3, and a message is published to NATS to notify the system that the "Abu Al-Hussain" record is successfully processed and ready for the UI.

Here is a deep dive specifically into how the Edge Case Engine runs under the hood.

The edge case flow is designed to be additive and non-blocking. Because querying an LLM specifically for edge-case name permutations takes time, the system is built using asynchronous Python threads (concurrent.futures) so it doesn't slow down the main data extraction pipeline.

Here is the exact mechanics of how it executes:

1. The Handoff (Spawning the Thread)
When an entity passes the similarity gate (score >= 0.90), the system invokes run_edge_case_flow_parallel().

What it does: Instead of executing the logic immediately, this function creates a ThreadPoolExecutor(max_workers=1).
It packages the task (run_edge_case_flow) along with the entity's complex_name, primary_name, and context, and submits it to this executor.
The Return: It immediately returns a Future object to the main loop. A Future is essentially a placeholder representing the result of a background operation that hasn't finished yet.
Result: The main extraction thread instantly moves on to process the next entity in the document while the edge-case thread starts doing the heavy lifting in the background.
2. The Background Execution (run_edge_case_flow)
While the main pipeline continues, the background thread executes the following sequence:

Reference Loading: It loads the edge_case_reference.json file.
Top-X Search: It runs a similarity search (find_top_x_similar) to find the top 5 (by default) closest matching complex names from the reference data.
Prompt Construction: It takes the entity's data and the 5 reference examples and feeds them into edge_case_generation_prompt(). This builds a highly specific few-shot prompt for the LLM.
LLM Execution: It makes a network call to the LLM (using llm_call_and_parse_json_dict_response). The LLM does the complex reasoning to generate structural, phonetic, and non-Latin permutations.
Return Data: The thread finishes and stores a dictionary containing the new aliases, aliases_og, and its reasoning inside the Future object.
3. Synchronization and The Safety Net (Timeouts)
Once the main thread has finished looping through all the entities and spawning background threads for the complex ones, it enters Phase 2.

Now, the main thread needs to collect the results from those background tasks before moving to the final database/S3 upload.

Collecting Futures: The system iterates over the map of Futures it collected earlier.
The 120-Second Safety Net: For each future, it calls future.result(timeout=120). This tells the main thread: "Wait here for the background thread to finish, but if it takes longer than 120 seconds, give up."
Timeout Handling: If the LLM hangs or the network call fails and hits the 120-second timeout, the system throws a TimeoutError. However, it does not crash the pipeline. It simply logs an error ("Gate future timed out...") and skips adding edge-case aliases for that specific entity. The entity still proceeds downstream with its standard aliases intact.
Successful Merge: If the future completes within the time limit, the data is pulled out and sent to the _merge_and_dedup function to be appended to the entity's main profile.
Summary of the execution model:
It acts as a scatter-gather architecture.

Scatter: Iterate through entities; if complex, scatter the work to a background thread.
Execute: Background threads independently call the LLM.
Gather: Main thread waits up to 2 minutes to gather the results back together, dropping any tasks that hang, before finalizing the document.
